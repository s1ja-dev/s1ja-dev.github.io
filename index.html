-- client sends an item id. that's it.
local function onRequestBuyItem(player, itemId)
  if typeof(itemId) ~= "string" then return end

  local item = ItemConfig.GetById(itemId)
  if not item then return end

  if profile.Currency.Coins < item.ShopPriceCoins then
    firePurchaseFailed(player, "NotEnoughCoins")
    return
  end
  if not InventoryService.HasCapacity(player, itemId, 1) then
    firePurchaseFailed(player, "InventoryFull")
    return
  end

  profile.Currency.Coins -= item.ShopPriceCoins
  InventoryService.AddItem(player, itemId, 1)
end

how i work

The same rules on every project

Not slogans — every one of these is enforced in the code linked above.

01

Server-authoritative by default

The client sends intent, never numbers. Damage, prices, RNG outcomes, and round state are decided and owned server-side.

02

Idempotent monetization

Every Robux purchase is checked against a persisted ledger before granting — safe under Roblox's automatic ProcessReceipt retries.

03

Defensive DataStore

pcall + backoff on every read/write, staggered autosaves, a real save budget on BindToClose. Data loss is a bug class, not an edge case.

04

Typed, strict Luau

--!strict everywhere, shared Signal/Trove/Class primitives reused across projects instead of reinvented per game.

05

One-directional data flow

Remotes → state → UI render. The client never mutates game state directly.

06

Verified before it's called done

Every system was exercised live in Play mode — distribution checks, timing checks, full state-machine walkthroughs.
